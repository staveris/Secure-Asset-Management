A.3 Pure engine — server/nis2-applicability.ts

Mirror dora-applicability.ts (no DB/Express imports; return a reason everywhere).

Regulatory logic to implement (NIS2 Directive (EU) 2022/2555):


deriveSizeClass(profile) → MICRO/SMALL/MEDIUM/LARGE using the EU SME definition:

MICRO: < 10 staff and ≤ €2M turnover and ≤ €2M balance sheet.
SMALL: < 50 staff and ≤ €10M turnover and ≤ €10M balance sheet.
MEDIUM: < 250 staff and (≤ €50M turnover or ≤ €43M balance sheet).
LARGE: otherwise.
(Use the standard "and for the ceilings, or between turnover/balance sheet" reading. If inputs are missing, return null and treat size as indeterminate — see rule 3.)



decideNis2Applicability(profile) → { inScope: boolean, entityClass: "ESSENTIAL" | "IMPORTANT" | null, sizeClass, reason: string }. Precedence, top to bottom:

If explicitlyExcludedByMemberState → not in scope, reason "Explicitly excluded by Member State".
If adminOverrideEnabled → in scope, entityClass = adminOverrideEntityClass ?? computed, reason "Manual admin override".
If !nis2ScopeConfirmed → not in scope (undetermined), reason "Scope not confirmed".
If !establishedInEuEea and not memberStateDesignatedInScope → not in scope, reason "Not established in EU/EEA and not designated".
Sector gate: if sectorGroup/sector is not a recognised Annex I or II sector (validate against NIS2_SECTORS) and not memberStateDesignatedInScope → not in scope, reason "Sector not in Annex I or II".
Size gate with size-independent carve-outs:

In scope regardless of size if any of: sizeIndependentEntity (DNS providers, TLD name registries, trust service providers, providers of public electronic communications networks/services, top-level registrars), publicAdministrationEntity, soleProviderInMemberState, memberStateDesignatedInScope.
Otherwise require MEDIUM or LARGE size. If MICRO/SMALL and no carve-out → not in scope, reason "Below size threshold (micro/small) and no size-independent trigger".
If size indeterminate (missing inputs) and no carve-out → in scope = false with undetermined note, reason "Size undetermined — provide headcount/turnover to complete scoping". (Do not guess.)



Essential vs Important classification (when in scope):

ESSENTIAL if: LARGE entity in an Annex I sector; or any size-independent essential type (DNS/TLD/trust/public-comms per the Directive); or central public administration; or the entity qualifies as essential under a Member-State designation flag.
IMPORTANT otherwise (e.g. MEDIUM Annex I entities, and Annex II entities not otherwise essential).
Always populate reason with the deciding facts (sector group, size class, trigger).






getControlTags(ctrl) — reuse the same accessor shape as DORA (reads ctrl.applicability.tags, returns string[]).
isControlApplicable(profile, ctrl) → { applicable, reason }:

If !decideNis2Applicability(profile).inScope → not applicable, propagate the reason.
Tag gates (a control applies if it matches the entity’s class/sector; see tag taxonomy A.4):

ALL_NIS2_ENTITIES → always applies when in scope.
ESSENTIAL_ONLY → applies only if entityClass === "ESSENTIAL" (e.g. ex-ante supervision, mandatory audits).
IMPORTANT_ONLY → applies only if entityClass === "IMPORTANT" (ex-post supervision specifics).
Sector tags (e.g. SECTOR_DIGITAL_INFRASTRUCTURE, SUBSECTOR_DNS_PROVIDER) → apply only if the tenant’s sector/subsector matches.
SIZE_INDEPENDENT_ONLY → applies only if a size-independent trigger is set.



A control with no class/sector tag defaults to applicable-when-in-scope (backwards compatible with today’s ALL_NIS2_ENTITIES default).



computeApplicableNis2Controls(profile, controls) — filter helper mirroring computeApplicableDoraControls.

PHASE B — Cross-Framework Mapping ("Answer once, satisfy many")

Goal. Let evidence and answers captured once flow as suggestions to equivalent controls in other
frameworks (NIS2 ↔ DORA ↔ CIR internally, plus outbound mapping to ISO/IEC 27001:2022 Annex A and
NIST CSF 2.0 as reference targets), and give subscribers a coverage matrix showing how much of one
framework is already satisfiable from what they’ve answered elsewhere.


Compliance-integrity rule (highest priority for this phase): cross-framework propagation is
advisory only. Nothing is ever auto-marked implemented/compliant. Every propagated answer is a
PENDING suggestion that a full-access human must explicitly accept, and every acceptance is
audit-logged with the complete provenance chain (source control → crosswalk → target control). Only
EQUIVALENT relationships may pre-fill a status at equal strength; SUPERSET/SUBSET/PARTIAL may
suggest but must down-rank confidence and must not pre-fill a terminal "compliant" status.

B.4 Pure engine — server/cross-framework.ts (no DB/Express imports)


type SavedResponseFacts = { atomicControlId, implementationStatus, maturityLevel, confidence }.
planSuggestions(sourceFacts, crosswalks, targetSlots) → returns candidate suggestion objects. Rules:

Only generate a suggestion when the source response is meaningfully positive (e.g. status is
IMPLEMENTED or maturity ≥ a threshold constant) — never propagate "not started".
EQUIVALENT (confidence ≥ 80): may suggest the same status, maturity, and confidence.
SUPERSET (source covers more than target): may suggest status but cap suggested maturity/confidence one level below source; never a terminal "fully implemented" without human review.
SUBSET / PARTIAL: suggest only as "review — partially covered", never pre-fill an affirmative status; set suggestedStatus to a non-terminal value and low confidence.
RELATED: informational only; do not create an actionable suggestion (surface in coverage matrix only).
Every candidate carries a reason string with the provenance chain and relationship.



computeCoverage(answeredControlIds, crosswalks, frameworkKey) → per-target-framework { mappable, alreadyAnswered, potentialFromMapping, percentages } for the matrix.
Unit-test this file heavily (server/cross-framework.test.ts): positive/negative propagation, each relationship’s down-ranking, idempotent suggestion identity, coverage math.
