You are a senior security-focused full-stack engineer building a secure multi-tenant compliance SaaS:
“NIS2 Readiness Platform”.

PRIMARY GOALS
1) Provide article-level NIS2 readiness assessment, covering all entity obligations and enabling “audit-grade” traceability.
2) Provide deep-dive assessments that adapt to ALL NIS2 Annex I (high criticality) and Annex II (other critical) sectors/subsectors.
3) Provide progress tracking (gaps → action plan → evidence → verification → monitoring).
4) Provide PLATFORM_ADMIN aggregated statistics across all tenants WITHOUT data leakage.

NON-NEGOTIABLE SECURITY REQUIREMENTS
- Multi-tenant isolation is enforced server-side on every request.
- Defense-in-depth: application checks + database constraints + optional Postgres RLS.
- Encrypt evidence files at rest with envelope encryption (per-tenant DEK).
- Evidence locking (immutable after verification unless privileged + audited unlock).
- Strong authentication, session security, audit logging, and OWASP hardening.

TECH STACK (Replit-friendly)
- Next.js 14+ App Router, TypeScript

- Auth: custom session auth (HttpOnly cookies) OR NextAuth Credentials (but still implement custom session table for revocation)

STRICT TENANT ISOLATION
Every tenant-scoped table has tenantId.
All reads/writes MUST include tenantId filters unless PLATFORM_ADMIN.
Implement central guards:
- requireAuth()
- requireRole(...)
- requireTenantScope(requestedTenantId)
- assertResourceTenant(resource.tenantId)
Never trust client-provided tenantId—derive tenantId from session.

OPTIONAL Postgres RLS (defense-in-depth):
- Enable RLS on tenant-scoped tables.
- Set session variables per request transaction (e.g., app.tenant_id, app.role).
- Create policies that restrict access by tenant_id unless role=PLATFORM_ADMIN.

ANALYTICS (SNAPSHOT-BASED SERVICE) — MUST IMPLEMENT
Goal: Admin aggregated analytics without scanning raw tables constantly and without leakage.
Create:
- TenantDailySnapshot(id, tenantId, date,
  compliancePct, verifiedPct, maturityAvg, overdueTasks, openIncidents, evidenceCoveragePct,
  lastComputedAt)

Computation strategy:
- On any write to ControlImplementation/Task/EvidenceItem/IncidentCase/AssessmentSubmission, enqueue a lightweight “recompute snapshot for tenant” job.
- Also run a daily scheduled recomputation to fix drift.

Implementation:
- Use a simple in-app job runner (setInterval) for Replit OR a queue (BullMQ) if feasible.
- Snapshot service lives in /lib/analytics/snapshots.ts with idempotent recomputeTenantSnapshot(tenantId, date).

Admin aggregated analytics MUST read from TenantDailySnapshot:
- Global totals: tenants, active users, avg compliance, distribution of verifiedPct, overdue tasks, evidence coverage.
- Breakdown by sector/subsector, BUT apply k-anonymity: do NOT show group breakdown if groupCount < 5 (configurable).
- Drill-down requires explicit tenant selection and writes an AuditLog entry.

EVIDENCE ENCRYPTION (ENVELOPE ENCRYPTION) — MUST IMPLEMENT
- KEK is APP_KEK env var (32 bytes base64) kept outside DB.
- Each tenant has a DEK stored wrapped (encrypted) by KEK in TenantKey.dekWrappedBase64.
- On upload:
  - unwrap DEK using KEK
  - generate random fileKey (or reuse DEK directly; prefer per-file key wrapped by DEK if time permits)
  - encrypt file bytes with AES-256-GCM
  - store iv/tag and metadata in encryptionMeta
- On download:
  - authorization check
  - decrypt streaming server-side
- Provide /lib/crypto/envelope.ts with: getTenantDEK(), encryptBuffer(), decryptStream()

Evidence locking:
- When a control is marked VERIFIED, require at least one linked EvidenceItem; allow “Lock all linked evidence”.
- Locked evidence cannot be replaced/deleted by tenant users.
- Unlock requires EvidenceUnlockRequest approval by TENANT_ADMIN (or PLATFORM_ADMIN) and is fully audited.

ASSESSMENT ENGINE — MUST BE DEEPER AND SECTOR-AWARE
Implement “layered” assessment packs:
A) CORE NIS2 PACK (applies to all tenants)
- Governance/management accountability (Art 20) as multiple objectives (approval, oversight, training, accountability evidence)
- Risk management measures (Art 21): implement each minimum measure as separate ControlObjectives
- Incident reporting (Art 23): significance determination, playbooks, 24h/72h/1-month readiness
- Supply chain considerations (Art 21 + Art 22) as separate objectives
- Audit/supervision evidence readiness (requests for info, audits, scans) as an “audit pack”

B) SECTOR PACKS (deep dive per Annex sector/subsector)
For EACH sector/subsector, create additional ControlObjectives that tailor Art 21 to sector-specific realities.
Examples (not exhaustive; implement at least 20–50 extra objectives per sector pack):
- Energy (electricity/oil/gas/hydrogen): OT/ICS governance, segmentation, safety impact analysis, remote access controls, secure maintenance, backup/restore drills for SCADA, incident coordination with operations
- Transport (air/rail/water/road): safety-critical systems, operational continuity, third-party maintenance controls, segmentation and monitoring of operational networks
- Drinking water / waste water: OT/ICS, monitoring integrity, resilient operations, emergency procedures
- Health: clinical system resilience, medical device security processes, patch windows, data integrity and continuity
- Digital infrastructure (cloud/data center/CDN/DNS/TLD/trust services/e-comms): identity assurance, secure service management, tenant isolation, logging, DDoS preparedness, key management, incident comms
- ICT service management (MSP/MSSP): customer segregation, privileged access mgmt, secure tooling, incident notification obligations to customers, logging retention
- Public administration: baseline hardening, identity, continuity, supplier governance, secure communications
- Space ground infrastructure: high-availability, supply chain dependencies, secure uplink/ground ops procedures
- Manufacturing/chemicals/food/waste/postal: production continuity, supplier criticality, industrial network protections, asset inventory completeness
- Digital providers: platform security, abuse monitoring, resilience

Mechanics:

Assessment UX:
