Non-negotiable engineering guardrails (all phases)

PHASE A — Public Scope Check + Lead Report

Goal. A public, unauthenticated page where a visitor answers ~9 questions and instantly sees
the engine verdict (in scope?, ESSENTIAL/IMPORTANT, written reason). The full detail — per-profile
control counts, supervisory-regime notes, Art. 23 deadlines, cross-framework teaser — is unlocked
by submitting an email, which stores a lead and emails a link to a shareable, print-optimized
report page.

A.1 Schema — leads table (the only persistence on the public surface)

tsexport const scopeCheckLeads = pgTable("scope_check_leads", {
  id: serial("id").primaryKey(),
  email: text("email").notNull(),
  reportToken: text("report_token").notNull().unique(),   // 32+ bytes, crypto-random, URL-safe
  // The submitted answers + computed verdict, denormalized as jsonb so the report
  // is reproducible even if the engine evolves:
  answers: jsonb("answers").notNull(),                    // the sanitized wizard inputs
  verdict: jsonb("verdict").notNull(),                    // { inScope, entityClass, sizeClass, reason }
  controlStats: jsonb("control_stats"),                   // { applicable, excluded, byTag } computed at submit time
  consentText: text("consent_text").notNull(),            // exact consent string shown at capture time
  consentMarketing: boolean("consent_marketing").notNull().default(false),
  createdAt: timestamp("created_at").defaultNow().notNull(),
  convertedTenantId: integer("converted_tenant_id").references(() => tenants.id), // set by Phase B
  convertedAt: timestamp("converted_at"),
  deletedAt: timestamp("deleted_at"),                     // soft-delete for GDPR erasure
});

Run npm run db:push; confirm additive-only.

A.3 Public API (no auth middleware; mounted before session-requiring guards is NOT needed — just don't attach requireAuth)

POST /api/public/scope-check (scopeCheckLimiter)


Zod schema for exactly the public inputs: sectorGroup (ANNEX_I|ANNEX_II|NONE), sector, subsector?, country, employeeCount, annualTurnoverMeur, balanceSheetMeur, plus booleans sizeIndependentEntity, publicAdministrationEntity, soleProviderInMemberState, memberStateDesignatedInScope, explicitlyExcludedByMemberState. Validate sector/subsector against NIS2_SECTORS; coerce numerics; reject unknown keys (.strict()).
Build a Partial<Nis2RegulatoryProfile> with nis2ScopeConfirmed: true, establishedInEuEea: true (the public funnel targets EU entities; expose an establishedInEuEea question if trivial) and call decideNis2Applicability.
Compute control stats without leaking control content: load NIS2 atomic controls via the existing storage getter, run computeApplicableNis2Controls, and return only aggregate counts { applicable, excluded, total, excludedByReasonGroup }.
Response: { verdict: { inScope, entityClass, sizeClass, reason }, stats, disclaimer }. Nothing persisted. No cookies set.


POST /api/public/scope-check/report (scopeReportLimiter)


Body: the same answers object plus email (validated), consent: true (must be literal true), consentMarketing?: boolean.
Recompute the verdict server-side from the submitted answers (never trust a client-supplied verdict), generate reportToken with crypto.randomBytes(32).toString("base64url"), insert the lead, and send the report email via sendGenericEmail with link ${getAppBaseUrl()}/scope-report/${token}.
Always respond { ok: true, message: "If the address is valid, the report link has been sent." } — even on email-send failure (log it via logSecurityEvent-style console reporting; do not reveal failures).


GET /api/public/scope-report/:token (reportViewLimiter)


Look up by token where deletedAt IS NULL. Return the stored answers, verdict, controlStats, createdAt. 404 on unknown/deleted token. No email echoed back in the payload (the person viewing may have been forwarded the link).


POST /api/public/scope-report/:token/delete (scopeReportLimiter)


GDPR erasure: soft-deletes the lead (sets deletedAt, blanks email to "erased"). Link this from the report footer ("Delete my data").

PHASE C — Plan Tiers & Upgrade Walls

Goal. A planTier on tenants (FREE / STARTER / PROFESSIONAL / PARTNER) that maps to
feature-flag bundles plus numeric caps, with three deliberate walls: NIS2 response cap on FREE,
evidence upload gated to STARTER+, cross-framework acceptance gated to PROFESSIONAL+. Billing
checkout is out of scope; platform admins set tiers.

C.1 Pure tier module — shared/plan-tiers.ts (no DB/Express imports)

tsexport const PLAN_TIERS = ["FREE", "STARTER", "PROFESSIONAL", "PARTNER"] as const;
export type PlanTier = typeof PLAN_TIERS[number];

export const TIER_LIMITS: Record<PlanTier, {
  nis2ResponseCap: number | null;        // FREE: 25; others: null (unlimited)
  evidenceUpload: boolean;               // STARTER+
  crossFrameworkAccept: boolean;         // PROFESSIONAL+
  flagBundle: string[];                  // feature-flag keys auto-enabled at this tier
}> = { ... };

export function canSubmitNis2Response(tier: PlanTier, currentCount: number): { allowed: boolean; reason?: string };
export function tierAllows(tier: PlanTier, capability: "evidenceUpload" | "crossFrameworkAccept"): boolean;

Unit-test exhaustively (shared/plan-tiers.test.ts or server-side test file, matching existing Vitest layout): cap boundary (24 allowed, 25th blocked with reason), each capability per tier, unknown tier defaults to FREE-equivalent (fail closed).
